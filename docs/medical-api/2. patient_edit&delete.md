# 환자 정보 수정·삭제 API 명세

## 7.4 환자 정보 수정

### 요구사항

- 요구사항 ID: `REQ-PTNT-004`
- 로그인한 사내 개발진, 의료 실무진, 연구진은 환자의 이름과 연락처를 수정할 수 있다.
- 환자의 나이와 성별은 수정할 수 없다.
- 일부 필드만 전달하여 수정할 수 있는 partial update 방식으로 처리한다.

### Endpoint

```http
PATCH /api/v1/patients/{patient_id}
```

### 권한

- `STAFF`, `ADMIN`만 접근할 수 있다.
- `PENDING` 사용자는 접근할 수 없다.
- 부서와 관계없이 `STAFF` 또는 `ADMIN`이면 수정할 수 있다.
- 공통 dependency인 `require_staff_or_admin()`을 사용한다.

### Path Parameter

| 이름 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| `patient_id` | int | Y | 수정할 환자의 고유 ID |

### Request Header

```http
Authorization: Bearer {access_token}
Content-Type: application/json
```

### Request Body

| 필드 | 타입 | 필수 | 검증 규칙 |
| --- | --- | --- | --- |
| `name` | string | N | 앞뒤 공백 제거 후 1~50자 |
| `phone` | string | N | 숫자와 하이픈만 허용하며 저장 시 숫자만 유지 |

- `name`, `phone` 중 최소 하나는 전달해야 한다.
- 두 필드가 모두 누락되면 `422 Unprocessable Entity`를 반환한다.
- `age`, `gender` 등 수정이 허용되지 않은 필드는 받을 수 없다.
- 전달되지 않은 필드는 기존 값을 유지한다.

### 요청 예시: 이름과 연락처 수정

```json
{
  "name": "홍길순",
  "phone": "010-9999-0000"
}
```

### 요청 예시: 이름만 수정

```json
{
  "name": "홍길순"
}
```

### 요청 예시: 연락처만 수정

```json
{
  "phone": "010-9999-0000"
}
```

### 처리 규칙

1. JWT 인증 정보를 확인한다.
2. `require_staff_or_admin()`으로 사용자 권한을 검사한다.
3. `get_patient_or_404()`로 환자 존재 여부를 확인한다.
4. `name`, `phone` 중 하나 이상 전달되었는지 확인한다.
5. 이름이 전달된 경우 앞뒤 공백을 제거하고 길이를 검증한다.
6. 연락처가 전달된 경우 형식을 검증하고 `normalize_phone()`을 적용한다.
7. 전달된 필드만 수정한다.
8. `updated_at`을 현재 UTC 시각으로 갱신한다.
9. 수정된 환자 정보를 반환한다.

### 성공 응답

```http
200 OK
```

```json
{
  "id": 1,
  "name": "홍길순",
  "age": 35,
  "gender": "M",
  "phone": "01099990000",
  "created_at": "2026-07-21T07:00:00Z",
  "updated_at": "2026-07-21T08:30:00Z"
}
```

입력 연락처에 하이픈이 포함되어 있어도 저장값과 응답값은 숫자로만 구성한다.

### 실패 응답

#### 인증되지 않은 사용자

```http
401 Unauthorized
```

```json
{
  "code": "UNAUTHORIZED",
  "message": "Authentication required"
}
```

#### 수정 권한이 없는 사용자

```http
403 Forbidden
```

```json
{
  "code": "FORBIDDEN",
  "message": "Permission denied"
}
```

#### 존재하지 않는 환자

```http
404 Not Found
```

```json
{
  "code": "PATIENT_NOT_FOUND",
  "message": "Patient not found"
}
```

#### 수정할 필드가 없는 경우

요청:

```json
{}
```

응답:

```http
422 Unprocessable Entity
```

```json
{
  "code": "EMPTY_UPDATE_PAYLOAD",
  "message": "At least one field is required"
}
```

#### 입력값 검증 실패

```http
422 Unprocessable Entity
```

발생 조건:

- 이름이 빈 문자열 또는 공백으로만 구성된 경우
- 이름이 50자를 초과한 경우
- 연락처에 숫자와 하이픈 이외의 문자가 포함된 경우
- 허용되지 않은 필드가 전달된 경우

### 상태 코드

| 상태 코드 | 설명 |
| --- | --- |
| `200 OK` | 환자 정보 수정 성공 |
| `401 Unauthorized` | 인증 실패 |
| `403 Forbidden` | 수정 권한 없음 |
| `404 Not Found` | 환자가 존재하지 않음 |
| `422 Unprocessable Entity` | 요청 필드 또는 입력값 검증 실패 |

## 7.5 환자 정보 삭제

### 요구사항

- 요구사항 ID: `REQ-PTNT-005`
- 로그인한 사내 개발진, 의료 실무진, 연구진은 환자 정보를 삭제할 수 있다.
- 환자 삭제 시 해당 환자와 연결된 진료기록과 X-Ray 이미지도 함께 영구 삭제한다.
- 삭제 확인 모달과 팝업은 클라이언트에서 처리하며, 최종 확인 후 삭제 API를 호출한다.

### Endpoint

```http
DELETE /api/v1/patients/{patient_id}
```

### 권한

- `STAFF`, `ADMIN`만 접근할 수 있다.
- `PENDING` 사용자는 접근할 수 없다.
- 부서와 관계없이 `STAFF` 또는 `ADMIN`이면 삭제할 수 있다.
- 공통 dependency인 `require_staff_or_admin()`을 사용한다.

### Path Parameter

| 이름 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| `patient_id` | int | Y | 삭제할 환자의 고유 ID |

### Request Header

```http
Authorization: Bearer {access_token}
```

### Request Body

없음.

```text
DELETE 요청에는 별도의 Request Body를 전달하지 않는다.
```

Notion 요구사항의 `확인했습니다` 버튼은 프런트엔드의 최종 확인 절차이며, 버튼을 클릭하면 이 API를 호출한다.

### 삭제 대상

환자 삭제 시 다음 데이터를 삭제한다.

1. `patients` 테이블의 환자 정보
2. 환자와 연결된 `medical_records` 데이터
3. 진료기록과 연결된 `xray_images` 데이터
4. 서버 로컬 저장소에 저장된 실제 X-Ray 이미지 파일

### 처리 규칙

1. JWT 인증 정보를 확인한다.
2. `require_staff_or_admin()`으로 사용자 권한을 검사한다.
3. `get_patient_or_404()`로 환자 존재 여부를 확인한다.
4. 환자에게 연결된 X-Ray 이미지의 상대 경로 목록을 조회한다.
5. DB transaction 안에서 환자와 연관 데이터를 삭제한다.
6. DB cascade를 통해 연결된 진료기록과 X-Ray 이미지 DB 레코드를 삭제한다.
7. DB transaction을 commit한다.
8. 수집한 경로를 사용하여 `delete_xray_file()`로 실제 이미지 파일을 삭제한다.
9. 응답 본문 없이 `204 No Content`를 반환한다.

### X-Ray 파일 삭제 규칙

- DB에는 X-Ray 이미지의 상대 경로가 저장되어 있다.
- 삭제 전에 해당 환자와 연결된 모든 X-Ray 이미지 경로를 조회한다.
- DB 삭제가 성공한 후 실제 로컬 파일을 삭제한다.
- 파일이 이미 존재하지 않아도 환자 삭제는 성공으로 처리한다.
- 파일 경로를 클라이언트에 노출하지 않는다.

### 성공 응답

```http
204 No Content
```

응답 본문은 없다.

```text
Response Body 없음
```

`204 No Content` 응답에는 JSON 데이터를 포함하지 않는다.

### 실패 응답

#### 인증되지 않은 사용자

```http
401 Unauthorized
```

```json
{
  "code": "UNAUTHORIZED",
  "message": "Authentication required"
}
```

#### 삭제 권한이 없는 사용자

```http
403 Forbidden
```

```json
{
  "code": "FORBIDDEN",
  "message": "Permission denied"
}
```

#### 존재하지 않는 환자

```http
404 Not Found
```

```json
{
  "code": "PATIENT_NOT_FOUND",
  "message": "Patient not found"
}
```

#### 삭제 처리 중 서버 오류

```http
500 Internal Server Error
```

```json
{
  "code": "PATIENT_DELETE_FAILED",
  "message": "Failed to delete patient"
}
```

DB 삭제 처리 중 오류가 발생하면 transaction을 rollback하고 환자 및 관련 데이터를 유지한다.

### 상태 코드

| 상태 코드 | 설명 |
| --- | --- |
| `204 No Content` | 환자와 관련 데이터 삭제 성공 |
| `401 Unauthorized` | 인증 실패 |
| `403 Forbidden` | 삭제 권한 없음 |
| `404 Not Found` | 환자가 존재하지 않음 |
| `422 Unprocessable Entity` | `patient_id` 형식이 올바르지 않음 |
| `500 Internal Server Error` | DB 삭제 처리 중 서버 오류 |

## 문서 반영 참고사항

첨부 설계 문서의 오류 응답 규칙에 따라 오류 응답은 FastAPI 기본 `detail` 형식이 아니라 `code`, `message` 필드를 사용한다.

```json
{
  "code": "PATIENT_NOT_FOUND",
  "message": "Patient not found"
}
```
