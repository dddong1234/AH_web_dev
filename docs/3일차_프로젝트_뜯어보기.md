# 3일차 프로젝트 뜯어보기

## 1. 프로젝트 개요

본 프로젝트는 FastAPI를 기반으로 AI 헬스케어 웹 서비스를 구현하기 위한 팀 프로젝트이다.

현재 프로젝트에는 다음 기술이 사용되고 있다.

- FastAPI
- Pydantic
- SQLAlchemy 2.x 및 비동기 SQLAlchemy
- MySQL 8.0
- asyncmy
- Alembic
- Docker Compose
- uv

프로젝트는 API, 비즈니스 로직, 데이터 접근 로직, 데이터베이스 모델을 분리하는 계층형 구조를 사용한다.

일반적인 API 요청 처리 흐름은 다음과 같다.

```text
클라이언트
    ↓ HTTP 요청
API Router
    ↓ 요청 데이터 검증
Pydantic Schema
    ↓ 기능 실행
Service
    ↓ 데이터 접근 요청
Repository
    ↓ SQL 실행
SQLAlchemy Model
    ↓
MySQL
```

각 계층을 분리하면 코드의 결합도를 낮출 수 있고 기능 수정과 테스트가 쉬워진다.

## 3. 각 디렉터리의 역할

### 3.1 `app/core/`

`app/core/`는 특정 기능에 종속되지 않는 애플리케이션 공통 설정을 관리한다.

현재 포함된 내용은 다음과 같다.

- 환경변수 설정
- 데이터베이스 접속 URL 생성
- SQLAlchemy 비동기 엔진 생성
- 비동기 DB 세션 관리
- 모든 DB 모델이 상속할 Base 클래스
- 여러 모델에서 재사용할 Mixin 클래스

### 3.2 `app/models/`

`app/models/`는 SQLAlchemy ORM 모델을 작성하는 디렉터리이다. ORM 모델은 Python 클래스와 데이터베이스 테이블을 연결한다.

모델에는 다음 내용을 작성한다.

- 테이블 이름
- 컬럼 이름과 자료형
- Primary Key와 Foreign Key
- Unique 및 Index 제약조건
- nullable 여부와 기본값
- 다른 모델과의 관계

테이블별로 파일을 분리하는 것이 좋다.

```text
app/models/
├── __init__.py
├── user.py
├── patient.py
└── health_record.py
```

`app/models/__init__.py`에서는 Alembic이 모델을 발견할 수 있도록 모든 모델 클래스를 import해야 한다.

### 3.3 `app/repositories/`

`app/repositories/`는 데이터베이스 접근 로직을 담당한다.

- 데이터 조회
- 데이터 생성
- 데이터 수정
- 데이터 삭제
- 검색 조건과 정렬 적용
- SQLAlchemy 쿼리 작성

Repository에는 비밀번호 정책이나 회원가입 가능 여부와 같은 비즈니스 규칙을 넣지 않고 데이터 저장과 조회에 집중한다.

### 3.4 `app/schemas/`

`app/schemas/`는 API 요청과 응답의 데이터 형식을 Pydantic 모델로 정의한다.

- 요청 데이터 검증
- 필수값과 선택값 구분
- 문자열 길이와 숫자 범위 검증
- 응답에 포함할 필드 제한
- ORM 객체를 API 응답으로 변환

현재 `practice_apis.py` 내부에 있는 `UserResponse`, `UserSignUpRequest`, `UserUpdate`는 실제 기능 구현 단계에서 `app/schemas/`로 분리하는 것이 좋다.

### 3.5 `app/services/`

`app/services/`는 애플리케이션의 비즈니스 로직을 담당한다.

- 회원가입 가능 여부 확인
- 이메일 중복 확인
- 비밀번호 암호화
- 여러 Repository 호출 조합
- 권한 확인
- 트랜잭션 흐름 제어

Service는 API와 Repository 사이에서 실제 업무 규칙을 실행한다.

### 3.6 `app/apis/`

`app/apis/`는 클라이언트의 HTTP 요청을 받는 FastAPI Router를 관리한다.

- URL 및 HTTP Method 정의
- 요청 및 응답 Schema 지정
- Dependency Injection
- DB 세션 또는 인증 사용자 전달
- Service 호출
- HTTP 상태 코드 반환

API 계층에서는 복잡한 쿼리나 비즈니스 규칙을 직접 구현하지 않고 Service에 위임하는 것이 좋다.

## 4. 주요 파일의 역할

### 4.1 `app/main.py`

`app/main.py`는 FastAPI 애플리케이션의 실행 진입점이다.

현재 다음 역할을 수행한다.

1. `FastAPI()` 객체 생성
2. `practice_router` 등록
3. 프로젝트 루트 경로 계산
4. `static`, `media` 디렉터리 생성
5. 정적 파일 경로 마운트
6. `/healthcheck` API 제공
7. 메인 HTML 파일 반환
8. 프론트엔드 SPA를 위한 catch-all 처리

기능별 Router가 추가되면 `app.main`에서 각각 등록해야 한다.

### 4.2 `app/core/config.py`

`config.py`는 환경변수와 애플리케이션 설정을 관리한다.

현재 `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`, `DB_NAME`을 정의하며 `.env` 파일에서 값을 읽는다.

```dotenv
DB_ROOT_PASSWORD=root-password
DB_USER=health-user
DB_PASSWORD=health-password
DB_HOST=mysql
DB_PORT=3306
DB_NAME=ai_health
```

실제 `.env` 파일에는 비밀번호가 포함되므로 Git에 커밋하지 않아야 한다. 팀 공유용으로는 실제 비밀번호가 없는 `.env.example`을 작성하는 것이 좋다.

### 4.3 `pyproject.toml`

`pyproject.toml`은 프로젝트 정보, Python 버전, 패키지 의존성을 관리한다.

현재 주요 의존성은 다음과 같다.

| 패키지 | 역할 |
|---|---|
| `fastapi[standard]` | FastAPI 웹 애플리케이션 구현 |
| `pydantic-settings` | 환경변수 및 설정 관리 |
| `sqlalchemy[asyncio]` | 비동기 ORM 및 SQL 실행 |
| `asyncmy` | MySQL 비동기 드라이버 |
| `alembic` | 데이터베이스 스키마 마이그레이션 |
| `cryptography` | 암호화 및 MySQL 인증 지원 |
| `uuid6` | UUIDv7 생성 |

새 패키지는 다음과 같이 추가한다.

```bash
uv add 패키지명
```

### 4.4 `uv.lock`

`uv.lock`은 실제로 설치되는 패키지와 하위 의존성의 정확한 버전을 기록한다. 팀원과 배포 환경에서 동일한 패키지 버전을 설치하게 해준다.

```bash
uv sync
```

`uv.lock`은 일반적으로 Git에 포함하고 직접 편집하지 않는다.

## 5. 데이터베이스 연결 설정

`app/core/db/databases.py`는 SQLAlchemy와 MySQL의 연결을 설정한다.

현재 비동기 MySQL 드라이버인 `asyncmy`를 사용한다.

```python
DATABASE_PREFIX = "mysql+asyncmy://"
```

DB URL의 형태는 다음과 같다.

```text
mysql+asyncmy://사용자:비밀번호@호스트:포트/DB이름
```

`async_engine`은 DB 연결 엔진이고, `AsyncSessionLocal`은 요청마다 사용할 비동기 세션을 생성한다. `Base`는 모든 SQLAlchemy 모델이 상속하는 기반 클래스이다.

`async_get_db()`는 FastAPI Dependency Injection을 통해 API에 DB 세션을 전달한다.

```python
async def async_get_db():
    async with AsyncSessionLocal() as db:
        yield db
```

## 6. SQLAlchemy 모델과 Alembic 마이그레이션

SQLAlchemy 모델은 `Base`와 필요한 Mixin을 상속해 작성한다.

```python
from sqlalchemy import String
from sqlalchemy.orm import Mapped, mapped_column

from app.core.db.databases import Base
from app.core.db.models import TimestampMixin, UUIDMixin


class User(Base, UUIDMixin, TimestampMixin):
    __tablename__ = "users"

    email: Mapped[str] = mapped_column(
        String(255),
        unique=True,
        nullable=False,
        index=True,
    )
    name: Mapped[str] = mapped_column(
        String(50),
        nullable=False,
    )
```

모델 작성 후 `app/models/__init__.py`에서 import해야 Alembic이 모델을 발견할 수 있다.

```python
from app.models.user import User

__all__ = ["User"]
```

마이그레이션 파일은 다음 명령으로 생성한다.

```bash
uv run alembic revision --autogenerate -m "create initial tables"
```

생성된 마이그레이션은 다음 명령으로 DB에 적용한다.

```bash
uv run alembic upgrade head
```

현재 상태와 이력은 다음과 같이 확인한다.

```bash
uv run alembic current
uv run alembic history
```

마지막 마이그레이션을 취소하려면 다음 명령을 사용한다.

```bash
uv run alembic downgrade -1
```

자동 생성된 마이그레이션 파일도 테이블, 컬럼, Foreign Key, Unique, Index, nullable 설정이 올바른지 직접 검토해야 한다.

## 7. API 구현 흐름

회원가입 API를 예시로 한 처리 흐름은 다음과 같다.

1. 클라이언트가 `POST /api/v1/users` 요청을 보낸다.
2. API Router가 요청을 받는다.
3. Pydantic Schema가 이메일, 비밀번호, 이름을 검증한다.
4. Service가 이메일 중복 확인과 비밀번호 암호화를 수행한다.
5. Repository가 SQLAlchemy를 이용하여 사용자를 저장한다.
6. SQLAlchemy Model이 MySQL 테이블과 연결된다.
7. 결과가 응답 Schema로 변환되어 클라이언트에 반환된다.

```text
HTTP 요청
  → API Router
  → Request Schema 검증
  → Service 비즈니스 로직
  → Repository DB 접근
  → SQLAlchemy Model
  → MySQL
  → Response Schema
  → HTTP 응답
```
